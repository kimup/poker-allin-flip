# Owner Tasks

Codexや実装側で進められる作業とは別に、オーナー側で決める/実行する必要がある作業です。

## 早めに決めること

1. 公開先を最終決定する

   基本方針はVercelです。PRごとのプレビューURLが欲しいので、このプロジェクトにはVercelが合っています。比較は [Hosting Comparison](hosting-comparison.md) にまとめています。

2. GitHubリポジトリ名を決める

   仮名は `poker-allin-flip` です。公開URLにも影響します。

3. 公開範囲を決める

   完全公開にするか、最初は限定的に見せるかを決めます。GitHub Pagesは基本的に公開リポジトリと相性が良いです。

4. 見た目の方向性を決める

   候補:

   - ポーカーテーブル風
   - シンプルな分析ツール風
   - スマホゲーム風

   2人で1画面を見る用途があるため、カードと勝敗が大きく見えるUIが合いそうです。

## GitHubでやること

- GitHub上に新しいリポジトリを作る
- `main` を公開用ブランチとして扱う
- `develop` を作業用ブランチとして作る
- `develop` から `main` へのPull Request運用にする
- 必要なら branch protection を設定する

## 公開サービスでやること

### GitHub Pagesの場合

- repository settings で Pages を有効にする
- 公開元を `main` にする
- Viteの `base` 設定が必要か確認する

### Vercelの場合

- VercelアカウントにGitHubリポジトリを接続する
- Production Branch を `main` にする
- `develop` やPull RequestのPreview URLが出ることを確認する

## 実装前に確認したいこと

- 初期UIの言語は日本語だけでよいか
- 1人モードの「運がいい」の指標は、まず勝率だけでよいか
- セッション履歴をブラウザに保存してよいか
- スマホ縦向き、スマホ横向き、PCのどれを最優先にするか

## 実装後に確認すること

- ランダムに配られたカードが重複していない
- 勝敗表示が直感的に分かる
- 2人で1画面を見たときにカードと勝者が読める
- `develop` ブランチをローカルで起動できる
- PRを `main` にマージしたら公開サイトが更新される
- READMEのアーキテクチャ図が実装内容と一致している
